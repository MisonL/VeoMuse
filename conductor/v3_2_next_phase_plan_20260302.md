# VeoMuse V3.2 下一阶段规划（2026-03-02）

## 1. 规划背景

当前项目已完成 V3.1 规划收口，基线事实如下：
- 已归档轨道：`26`
- 已完成任务：`174/174`
- 自动化测试：`134` 项（全绿）
- 生产部署：Docker + Nginx + Backend + Redis 已可运行

下一阶段目标不再是“补功能数量”，而是把系统推进到“可规模化运营”的产品质量层级。

## 2. V3.2 总目标（North Star）

- `NS-01` 关键主链路（注册/组织/工作区/协作/导出）端到端成功率 `>= 99.5%`
- `NS-02` 非 AI API（工作区、存储、策略）P95 响应时间 `<= 400ms`
- `NS-03` 前端关键任务完成效率：新用户首次完成“创建工作区 -> 生成 -> 导出”平均步骤 `<= 8`
- `NS-04` 生产可靠性：数据库损坏修复演练脚本每周可稳定通过
- `NS-05` 安全门禁：主分支持续保持 secrets 扫描、鉴权回归、多租户越权回归全绿

## 3. 轨道拆分（Tracks）

## Track A：前端信息架构重构与体验可理解性

### 目标
把“功能很多但不易理解”优化为“高密度能力 + 可学习交互”。

### 关键任务
- 拆分超大页面组件（重点：`ComparisonLab.tsx`、`App.tsx`）
- 建立统一布局协议（顶部主流程区 / 中央预览区 / 左右工具区）
- 增加“任务导向入口”与“新手引导流”（按创作目标引导功能）
- 补齐关键操作反馈（加载态、错误态、禁用态、恢复路径）
- 建立跨分辨率视觉回归基线（桌面优先，兼容移动端）

### 验收标准
- `1366x768` 与 `1920x1080` 下无重叠、无遮挡、不可点击热区
- 核心流程任务可在 UI 文案引导下独立完成（不依赖口头说明）
- 关键交互具备明确状态反馈与恢复入口

## Track B：生产可靠性与故障自愈强化

### 目标
把“能跑”提升为“长时间稳定运行，出现故障可自动恢复”。

### 关键任务
- DB 自愈链路升级：健康检查、自动修复、回收结果可视化、历史追踪
- 清理与存储策略统一：上传目录、清理策略、容量阈值、预警机制
- 增加故障注入演练（数据库、协作 WS、存储上传、模型路由）
- 统一服务级指标：成功率、失败分类、P95、重试命中率
- 发布“运维 Runbook”（故障定位与恢复 SOP）

### 验收标准
- 自动修复演练可重复通过，且有结构化报告
- 线上关键异常可在 5 分钟内定位到模块级根因
- 清理任务在自定义 `UPLOADS_PATH` 场景一致生效

## Track C：多租户治理与成本控制

### 目标
从“支持多租户”升级为“租户隔离可审计、成本可治理”。

### 关键任务
- 组织级/工作区级配额体系（请求量、存储量、并发量）
- 渠道成本计量（按 provider 记录调用成本、延迟、成功率）
- 预算与熔断策略（阈值告警、自动降级、策略回退）
- 审计导出（组织管理员可导出策略变更与关键操作日志）
- 越权回归测试加固（组织隔离、工作区隔离、存储隔离）

### 验收标准
- 任一组织不可读取/影响其他组织策略与数据
- 预算超过阈值后系统可自动执行降级路由
- 成本/策略报表可按组织时间范围查询与导出

## Track D：发布工程与质量门禁（E2E）

### 目标
建立“每次发布可验证、可回滚、可追溯”的工程闭环。

### 关键任务
- 增加浏览器级 E2E 用例（真实 UI 点击链路，不只 API 测试）
- 增加视觉回归与布局回归快照（关键页面、关键分辨率）
- 建立发布门禁流水线（单测 + 集成 + E2E + secrets + 构建）
- 定义灰度发布策略与回滚触发条件
- 发布后巡检脚本化（健康检查、关键链路 smoke）

### 验收标准
- 主流程 E2E 用例稳定通过率 `>= 99%`
- 每次发布均有可追溯的质量报告与回滚记录
- 关键页面视觉回归可自动识别布局退化

## 4. 里程碑（建议 4 个 Sprint）

- `Sprint 1`（第 1-2 周）：Track A/B 基础骨架
  - 完成 IA 草图、组件拆分方案、故障演练基线
- `Sprint 2`（第 3-4 周）：Track A 深化 + Track C 基础
  - 完成交互引导、配额与成本采集最小闭环
- `Sprint 3`（第 5-6 周）：Track B/C 深化
  - 完成自愈看板、预算策略、审计导出
- `Sprint 4`（第 7-8 周）：Track D 与发布门禁
  - 完成 E2E 门禁、灰度发布、收口验收

## 5. 第一批可立即开工任务（P0）

- [ ] 将 `ComparisonLab.tsx` 拆分为模式级容器 + 领域子组件（降低耦合）
- [ ] 建立“功能导览 / 新手引导”最小可用版本
- [ ] 增加浏览器级 E2E：注册 -> 组织 -> 工作区 -> 生成 -> 导出
- [ ] 增加配额模型（组织维度请求次数与存储量）
- [ ] 增加策略变更审计导出接口
- [ ] 补齐发布门禁：E2E + secrets + build + smoke

## 6. 风险与约束

- 超大组件拆分期间，存在前端行为回归风险，需要配套 UI 回归测试。
- 多租户配额与成本计量会引入额外存储写入，需要同步评估 DB 压力。
- E2E 引入后 CI 时间会增长，需并行化或按分层策略执行。
